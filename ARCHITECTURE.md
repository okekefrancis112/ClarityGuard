## System Overview

ClarityGuard is a static analysis security scanner for Clarity smart contracts. It is written in TypeScript, has **zero runtime dependencies**, and runs on Node.js.

The scanner operates as a four-stage pipeline:

```
Source Code (.clar)
       │
       ▼
┌─────────────┐
│   Parser     │   Tokenize → Parse → Classify → AST
└──────┬──────┘
       │  ClarityNode[]
       ▼
┌─────────────┐
│  Analyzer    │   Run all detectors against the AST
└──────┬──────┘
       │  Finding[]
       ▼
┌─────────────┐
│  Reporter    │   Format as terminal output or JSON
└──────┬──────┘
       │
       ▼
  Console / File
```

## Data Flow

### End-to-End Execution

```
CLI Input (process.argv)
    │
    ▼
index.ts ─── main()
    │
    ├── Parse CLI arguments (scan | detectors | --help | --version)
    ├── findClarityFiles(target) ─── Recursively discover .clar files
    │
    ▼
analyzer.ts ─── analyzeMultiple(files)
    │
    ├── For each file:
    │   ├── parser.ts ─── parse(source) → ClarityNode[]
    │   │
    │   └── For each detector in ALL_DETECTORS:
    │       └── detector.detect(ast, source) → Finding[]
    │
    ├── Sort findings by severity, then line number
    ├── Aggregate statistics
    │
    ▼
reporter.ts ─── formatTerminal(results) or formatJSON(results)
    │
    ▼
Console output + exit code (0 = clean, 1 = critical/high found)
```

### CLI Argument Parsing

```
clarityguard scan <path>          → Scan with terminal output
clarityguard scan <path> --json   → Scan with JSON output
clarityguard detectors            → List all available detectors
clarityguard --help | -h          → Show usage
clarityguard --version | -v       → Show version
```

**Exit codes:**
- `0` — No critical or high severity findings
- `1` — Critical or high severity findings detected (fails CI)

## Type System

### Parser Types

```typescript
// All possible AST node types
type ClarityNodeType =
  | "list"       // S-expression: (define-public ...)
  | "symbol"     // Identifiers: define-public, var-get, +
  | "integer"    // Signed integers: -50, 42
  | "uint"       // Unsigned integers: u100, u1000
  | "string"     // String literals: "hello"
  | "principal"  // Addresses: 'SXYZ123..., .contract-name
  | "bool"       // true | false
  | "keyword"    // Reserved
  | "response"   // Reserved
  | "optional"   // Reserved
  | "tuple-key"  // Tuple keys: name:, amount:

// AST node — the fundamental unit of analysis
interface ClarityNode {
  type: ClarityNodeType;
  value?: string | number | boolean;  // Leaf node value
  children?: ClarityNode[];           // List node children
  line: number;                       // 1-based line position
  column: number;                     // 1-based column position
}
```

### Internal Token Type (Parser)

```typescript
interface Token {
  type: "lparen" | "rparen" | "atom" | "string" | "comment";
  value: string;
  line: number;
  column: number;
}
```

### Detector Types

```typescript
// Five severity levels, ordered by impact
type Severity = "critical" | "high" | "medium" | "low" | "info"

// A single detected issue
interface Finding {
  detector: string;          // Detector ID: "unprotected-admin"
  severity: Severity;
  title: string;             // Short summary
  description: string;       // Detailed explanation
  line: number;              // Source line
  column: number;            // Source column
  snippet?: string;          // Optional code excerpt
  recommendation?: string;   // Remediation advice
}

// Contract for all detectors
interface Detector {
  id: string;
  name: string;
  severity: Severity;
  description: string;
  detect: (ast: ClarityNode[], source: string) => Finding[];
}
```

### Analysis Types

```typescript
interface AnalysisResult {
  file: string;
  findings: Finding[];
  stats: {
    totalFindings: number;
    critical: number;
    high: number;
    medium: number;
    low: number;
    info: number;
    detectorsRun: number;
    parseErrors: string[];
  };
}
```

## Parser

**File:** `src/parser.ts`

The parser converts raw Clarity source code into a typed AST using a two-phase approach.

### Phase 1: Tokenization

The `tokenize(source)` function performs lexical analysis, converting source text into a flat token stream.

**Algorithm:**
1. Iterate character-by-character, tracking line and column numbers
2. Skip whitespace (updating position counters)
3. Skip comments (`;` through end of line)
4. Match `(` and `)` as `lparen`/`rparen` tokens
5. Match string literals with escape character handling
6. Match atoms (contiguous non-whitespace, non-paren sequences)

### Phase 2: AST Construction

The `parseExpr()` function recursively builds the tree from the token stream:

```
lparen token → Create "list" node, recursively parse children until rparen
string token → Create "string" node
atom token   → Classify via classifyAtom()
```

### Atom Classification

The `classifyAtom(token)` function determines the node type from raw text:

| Pattern | Resulting Node Type |
|---------|-------------------|
| `true` / `false` | `bool` |
| `u\d+` (e.g., `u100`) | `uint` |
| `-?\d+` (e.g., `42`, `-5`) | `integer` |
| `'S...` or `'T...` | `principal` |
| `.contract-name` | `principal` |
| `word:` | `tuple-key` |
| Everything else | `symbol` |

### Utility Functions

| Function | Signature | Purpose |
|----------|-----------|---------|
| `getFunctionName` | `(node: ClarityNode) → string \| null` | Extract the function name from a list node (first child symbol) |
| `walkAST` | `(nodes, callback, parent?) → void` | Visitor-pattern traversal — calls `callback(node, parent)` for every node |
| `findDefinitions` | `(ast: ClarityNode[]) → ClarityNode[]` | Filter top-level nodes to `define-public`, `define-private`, `define-constant`, etc. |

### Parse Errors

The parser throws on:
- Unclosed parentheses
- Unexpected closing parentheses
- Unexpected end of input

These are caught by the analyzer and recorded in `stats.parseErrors`.

## Analyzer

**File:** `src/analyzer.ts`

The analyzer orchestrates detector execution and aggregates results.

### Detector Registry

All detectors are registered in a static array:

```typescript
const ALL_DETECTORS: Detector[] = [
  unprotectedAdmin,      // CRITICAL
  unsafeUnwrap,          // HIGH
  divisionBeforeMultiplication,  // MEDIUM
  blockHeightTiming,     // MEDIUM
  txSenderInAssert,      // MEDIUM
  hardcodedPrincipals,   // LOW
  uncheckedExternalCalls, // HIGH
  missingInputValidation, // MEDIUM
  deadCode,              // INFO
  todoComments,          // INFO
];
```

### Analysis Pipeline

**`analyze(source, filename) → AnalysisResult`**

1. Parse source into AST (catch parse errors)
2. Run each detector independently (catch individual failures)
3. Sort findings by severity (critical first), then by line number
4. Count findings per severity level
5. Return `AnalysisResult` with findings and stats

**`analyzeMultiple(files) → AnalysisResult[]`**

Maps each file through `analyze()` for batch processing.

**Key design decisions:**
- Every detector always runs (no early exit)
- Individual detector failures are non-fatal (logged to `parseErrors`)
- Findings are deterministically sorted for consistent output

## Detector System

Each detector implements the `Detector` interface: `(ast, source) → Finding[]`

### Detector Catalog

#### CRITICAL

**`unprotected-admin`** — Finds public functions that modify state without authorization checks.

- Scans for `define-public` functions
- Checks for state-modifying calls: `var-set`, `map-set`, `stx-transfer?`, `ft-mint?`, `nft-mint?`, etc.
- Checks for auth patterns: `tx-sender`, `contract-caller`
- Flags if state is modified but no auth check exists

#### HIGH

**`unsafe-unwrap`** — Detects `unwrap-panic` and `unwrap-err-panic` usage.

- Walks the AST for calls to panic-unwrap functions
- These abort transactions with no error information
- Recommends `unwrap!`, `try!`, or `match` instead

**`unchecked-external-calls`** — Finds `contract-call?` without response validation.

- Locates all `contract-call?` nodes
- Checks if the parent is a response handler (`try!`, `unwrap!`, `match`, etc.)
- Flags unhandled external call responses

#### MEDIUM

**`division-before-multiplication`** — Detects precision loss from `(* (/ a b) c)` patterns.

- Finds all `*` nodes in the AST
- Checks if any child is a `/` node
- Integer division before multiplication discards precision

**`block-height-timing`** — Warns about `block-height` in time-sensitive logic.

- Searches for `block-height`, `burn-block-height`, `stacks-block-height` symbols
- Block times are variable on Stacks; not suitable for precise timing
- Uses a `Set<number>` to deduplicate by line

**`tx-sender-in-assert`** — Flags `tx-sender` in authorization assertions.

- Finds `asserts!` / `assert!` calls
- Checks if the condition references `tx-sender`
- Recommends `contract-caller` to prevent relay attacks

**`missing-input-validation`** — Finds numeric parameters without bounds checks.

- Extracts parameters from `define-public` functions
- Checks if each numeric param appears in a comparison or assertion
- Validation functions: `>`, `<`, `>=`, `<=`, `is-eq`, `asserts!`

#### LOW

**`hardcoded-principals`** — Detects principal addresses outside of constants.

- **Pass 1:** Build an allowlist of principals inside `define-constant`
- **Pass 2:** Flag any principal node not in the allowlist

#### INFO

**`dead-code`** — Finds unreferenced private definitions.

- Collects all `define-private`, `define-read-only`, `define-constant`, `define-data-var` names
- Collects all symbol references across the AST (excluding the definition itself)
- If a name is never referenced, it's dead code

**`todo-comments`** — Matches `TODO`, `FIXME`, `HACK`, `XXX`, `BUG`, `WARNING` in comments.

- Line-by-line regex scan of the source text
- One finding per matching line

### Adding a New Detector

1. Create `src/detectors/my-detector.ts`:

```typescript
import { Detector, Finding } from './index';
import { ClarityNode, walkAST, getFunctionName } from '../parser';

export const myDetector: Detector = {
  id: 'my-detector',
  name: 'My Detector',
  severity: 'medium',
  description: 'Detects a specific pattern.',

  detect(ast: ClarityNode[], source: string): Finding[] {
    const findings: Finding[] = [];

    walkAST(ast, (node, parent) => {
      // Detection logic here
      if (/* condition */) {
        findings.push({
          detector: this.id,
          severity: this.severity,
          title: 'Issue found',
          description: 'Detailed explanation.',
          line: node.line,
          column: node.column,
          recommendation: 'How to fix it.',
        });
      }
    });

    return findings;
  }
};
```

2. Register in `src/analyzer.ts`:

```typescript
import { myDetector } from './detectors/my-detector';

const ALL_DETECTORS: Detector[] = [
  // ... existing detectors
  myDetector,
];
```

3. Add test cases in `test/samples/`.

## Design Patterns

### 1. Visitor Pattern — AST Traversal

```typescript
walkAST(ast, (node, parent) => {
  // Visit every node without managing recursion
});
```

Detectors use `walkAST` to traverse the tree without coupling to its structure.

### 2. Strategy Pattern — Pluggable Detectors

Each detector is an interchangeable strategy implementing the same interface. The analyzer invokes them uniformly:

```typescript
for (const detector of ALL_DETECTORS) {
  findings.push(...detector.detect(ast, source));
}
```

### 3. Registry Pattern — Detector Discovery

The `ALL_DETECTORS` array acts as a static registry. Adding a new detector requires only importing it and appending to the array.

### 4. Pipeline Pattern — Staged Processing

Data flows through discrete, composable stages:

```
Parse → Analyze → Report
```

Each stage has a clear input/output contract, making it testable in isolation.

## Error Handling

### Strategy

ClarityGuard uses a **fail-soft** approach: individual failures are isolated and logged, but the tool continues processing.

### Parse Errors

```typescript
try {
  ast = parse(source);
} catch (err) {
  // File is unparseable — record error, return empty findings
  parseErrors.push(err.message);
  return { file, findings: [], stats: { parseErrors, ... } };
}
```

Thrown on: unclosed parentheses, unexpected tokens, unexpected end of input.

### Detector Errors

```typescript
for (const detector of ALL_DETECTORS) {
  try {
    findings.push(...detector.detect(ast, source));
  } catch (err) {
    // Detector failure is non-fatal — other detectors still run
    parseErrors.push(`Detector "${detector.id}" failed: ${err.message}`);
  }
}
```

### File System Errors

```typescript
if (!fs.existsSync(target)) {
  console.error(`Error: '${target}' does not exist.`);
  process.exit(1);
}
```

Fails fast with a clear message before analysis begins.

## Constraints & Future Considerations

### Current Limitations

| Limitation | Impact | Mitigation Path |
|-----------|--------|-----------------|
| No parent pointers in AST | Some detectors need workarounds to check context | Add optional `parent` field to `ClarityNode` |
| No span tracking | Positions mark the start, not the full range | Extend parser to track end line/column |
| No semantic type checking | Can't validate types across function calls | Build a type inference pass |
| No control flow analysis | Can't track data across branches | Add CFG construction phase |
| No cross-file analysis | Each file is analyzed independently | Build a module resolver |

### Planned Expansions

- **10 additional detectors** (post-condition analysis, reentrancy patterns, flash loan vectors, etc.)
- **Web dashboard** (Next.js) for browser-based scanning
- **GitHub Action** for automated PR scanning
- **VS Code extension** for inline warnings
- **Property-based fuzzing** engine
